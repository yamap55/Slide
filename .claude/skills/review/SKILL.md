# 引数

このスキルは第一引数として、スライド原稿（`YYYYMMDD/slide.md` など）へのパスを受け取る。指示やコマンドにファイル名が含まれていない場合は、ユーザーに入力を促す。

# このスライドの前提（レビューの土台）

- 発表時間は短い（具体的な時間目安は AGENTS.md「スライド構成の方針」を参照）。
- 目的は「イメージを掴んでもらうこと」。詳細を覚えてもらうことではない。細部の割愛はむしろ正しい。
- テーマは技術に限らない。技術前提を聞き手に求めすぎていないかという観点で見る。
- フォーマットの正本は `AGENTS.md`、文章表現の正本は `writing` スキル。本スキルは、出来上がった原稿にそれらの観点で問題が出ていないかを検出する。

# レビューしてほしい観点

1. タイトル
   - 内容とズレていないか
   - 一覧（README）に並んだとき分かりやすいか
   - 不必要に強すぎる表現になっていないか

2. アジェンダ・導入
   - 冒頭にアジェンダがあり、全体像が先に見えるか
   - アジェンダの項目が、本文の `---` セクションと対応しているか
   - 「なぜこの話をするのか」が自然に伝わるか
   - 導入が長すぎたり説明過多になっていないか

3. 構成・流れ
   - 話の流れ（問い → 中身 → 着地）が分かりやすいか
   - 山場（オチ・結論・キーとなる数字や事実）を、そこに至る前のスライドで先に明かしてしまっていないか
   - 各スライドが自己完結しているか（他のスライドの補足を前提にしていないか）
   - 前提知識が暗黙になりすぎていないか。逆に当たり前すぎる説明が冗長になっていないか

4. 時間配分と情報量
   - 全体の分量が時間目安（AGENTS.md「スライド構成の方針」を参照）に収まりそうか。多すぎる場合は削る候補を挙げる
   - 1 スライドに詰め込みすぎていないか（箇条書きが多すぎる、1 項目が長すぎる、読み切れない密度）
   - 「イメージを掴んでもらう」目的に対し、細部に立ち入りすぎていないか

5. 内容の正確さ
   - 誤解を招く表現がないか
   - 曖昧に見える部分があれば指摘する
   - 根拠なく断定していないか。逆に、本物の不確実性を表す弱い述語が残っているか

6. 固有名詞・サービス名・ツール名の表記
   - 大文字・小文字・スペース・正式名称が正しく書かれているか
   - 人名・製品名・作品名・チーム名などの固有名詞が正式な表記になっているか（テーマが技術に限らないため、技術用語以外も対象）
   - 技術的な内容を含む場合、表記ゆれ・誤記がないか
     - Javascript → JavaScript
     - github → GitHub
     - GCP → Google Cloud
     - postgres / postgre → PostgreSQL
   - 聞き手に伝わらない専門用語を、説明なしに使っていないか

7. 出典・参考リンク
   - 引用・データ・仕様などに触れている場合、その取得元（公式サイト・公式ドキュメント・記事など）へのリンクがあるか
     - 「〇〇の仕様として〜」「ドキュメントに〜と書かれている」「データによると〜」のような記述
   - リンクは、本筋の途中ではなく出典・参考をまとめた 1 枚に置けているか（a タグ／Markdown リンクで、発表中は飛ばせる形）
   - Google / VS Code / Python など、誰でも分かるものはリンクなしで OK
   - 足りない場合は「ここに〇〇の公式リンクがあるとよい」と具体的に指摘する

8. 前提環境（コードや手順を含む場合）
   - コード（SQL / Python / JavaScript など）や手順がある場合、どのような環境・前提で書かれているかが分かるか
     - 言語・バージョン（例：Python 3.12、Node.js 20）
     - OS（Windows / Linux など）、DB・ミドルウェアの種類とバージョン（例：PostgreSQL 15）
   - すべてを厳密に書く必要はないが、前提が必要な場合は 1 枚にまとめて置けているか
   - 不足している場合は「この前提を 1 枚にまとめておくと親切」と指摘する

9. まとめ・締め
   - 全体の着地として違和感がないか
   - 「イメージを掴んでもらう」という目的に対して、聞き手に何が残るかがはっきりしているか

10. 文章表現
   - 執筆時の規範は `writing` スキルが正本。ここではその規範に照らして、出来上がった原稿に問題が出ていないかを検出する。`writing` スキルの「文章表現」節を改訂したら、この観点も合わせて見直す。
   - 太字（`**`）の多用がないか（目安は 1 スライドに一、二箇所まで）。箇条書き冒頭が機械的に太字化されていないか
   - LLM っぽい定型表現が混入していないか
     - 予告・総括（「重要なのは〜である」「本章では〜を扱う」「〜に他ならない」）
     - 空虚な形容（「不可欠」「核心的」「鍵となる」「多角的」「包括的」）
     - 空虚な動詞（「掘り下げる」「深掘りする」「言語化する」）
     - 空虚な緩和・称賛（「〜と言えるだろう」「非常に」「極めて」）
   - 論証の厳密さ
     - 複数要因を単一原因に還元していないか、異なる対象を一括りにしていないか
     - 因果を主張する箇所で、その機構が一言で示されているか
   - 冗長がないか（同じ主張の言い換え反復、場面描写直後の要約のし直し、接続や評価だけの行）
   - 見出しが具体的か（情報量のない見出し・結論を言い切る「セリフ」だけになっていないか）。観点 1 のタイトルと地続きで見る
   - 誠実さ（確認していないことを確認したかのように書いていないか。作為的に見える例を隠していないか）

11. reveal.js の構造（正本は AGENTS.md）
   - 先頭に CSS スタイル（見出しの大文字変換を無効化するもの）が置かれているか
   - 区切り文字が正しく使われているか（`---` 新しいスライド／`--` 同一スライド内の縦区切り）
   - 画像は相対パスで参照され、同じディレクトリに配置されているか
   - スピーカーノート（`Note:`）を使う場合、書式が正しいか

12. README への追加
   - スライド作成後、README.md の一覧の先頭付近に行が追加されているか
   - 日付の降順（新しい順）で正しい位置に入っているか
   - 表の形式（タイトル・日付・コードへのリンク・備考）に従っているか

# レビューのスタンス

- 全否定ではなく、改善提案ベースで行う。
- 「ここは良い」「この方向性は合っている」も言語化する。
- 修正するなら、どこをどう直すかを具体的に示す。
- 細部の割愛はこのスライドでは正しい。網羅性の不足を一律に欠点として指摘しない。
